01 — what it does

A composed document, not a transcript dump

Open the file in any browser and you get a synthesized document — a summary and type tags, architecture and flow diagrams as inline SVG, decision records (what was chosen, what was rejected, why), a file-touch list with real path:line references, and a timeline where the work has one.

For richer surfaces there are two interactive components: an interactive explorer — a selector that drives a live diagram region and a detail pane from one shared data model, for navigating relationships in a multi-component architecture rather than reading a static diagram — and a data-driven step timeline — phases of parallel and sequential steps with source pills and micro-cycles, for sessions that cover a pipeline, workflow, or multi-phase build.

Detail is clickable: expand a node or a decision to see the supporting context. Components adapt to the content — the catalog is a floor, not a ceiling. When something fits no off-the-shelf component, a bespoke one is composed from the same design-system primitives so it still looks native, and empty sections are omitted rather than stubbed. After writing the file, create surfaces a short coverage summary — what made it into the document and what was intentionally omitted — so you can spot anything left out before you close the session.

No setup, no config

Install and run. There is no setup step and no configuration file required — /walkthrough:create [focus] produces a readable artifact of the session whenever you want one: after a debugging marathon, at the end of a feature, before a handoff, or to share with someone who wasn't in the room.

04 — reference

Three skills, one shared renderer

All three are /walkthrough:<name>, all default frontmatter — user- and intent-invocable. Typing the slash form works; so does asking in plain language. There are no hidden building blocks, no agents, no hooks, no scripts.

SkillInputWhat it does
/walkthrough:createsession autoRenders the current session from scratch through the five-stage pipeline. The optional [focus] argument scopes the synthesis. Also fires on plain-language intent — "visualize this session", "walk me through what we did", "walk me through it" (or "this" / "that"), or "make a session recap".
/walkthrough:updatesession autoRefreshes an existing walkthrough in place — reconstructs the prior model from the rendered HTML, folds in the spec/source files you name, and overwrites the same file. Always asks which document first.
/walkthrough:documentsubject autoRenders a subject — a plugin, the marketplace, or any path — instead of a session. README + manifest are canonical. This is what builds every page under site/, including this one.

The tool that documents itself

Every page under site/ — the marketplace landing plus one per plugin — is generated by /walkthrough:document, never hand-written. This page is a rendering of walkthrough's own README + manifest. Change the README, regenerate the page.

05 — the core promise

Self-contained, tokens only

Two invariants make every walkthrough recognizably the same artifact and portable anywhere. They are non-negotiable.

self-contained
Everything inlined

All CSS, JS, and SVG live in the single HTML file. The only external resource is one Google Fonts @import — no <script src>, no <link rel=stylesheet>, no <img>, no CDN.

tokens only
One house style

Never a raw hex, font, or spacing literal — everything goes through CSS custom properties. Both themes are :root token blocks; the toggle swaps token sets, not stylesheets.

two themes
Dark + warm-light

One in-document toggle, viewer's choice per-view. Nothing about it requires a server or a build step.

open components
Floor, not ceiling

Catalog entries used verbatim where they fit; bespoke components composed from the same primitives where they don't — passing a looks-native checklist. A .chip primitive (five status roles: ok, info, warn, danger, neutral) drops a status badge inline in any component.

The result renders with no network beyond that one font fetch — copy the file anywhere, email it, drop it in a wiki, and it works without the plugin, without a server, and without a connection. Walkthroughs are written to .claude/walkthrough/<YYYY-MM-DD-HHMM>-<slug>.html; on first run create offers to add that directory to .gitignore, since a session artifact can contain code and prose you exchanged. When a walkthrough on the same subject already exists, a proliferation guard kicks in — create asks whether to update it in place, write a new versioned file, or overwrite, rather than silently piling up -2/-3 copies.

07 — boundaries

What it is — and isn't

A walkthrough is a snapshot, not a live view. Knowing the boundary keeps expectations honest.

it is
A single self-contained file

One HTML file with everything inlined. Renders without the plugin, without a server, and without a network beyond the font fetch.

it is
A synthesized model

Sections, diagram nodes, decisions, and files — rendered as a composed document, not pasted from the message log.

it isn't
A live viewer

It's a snapshot of one session at the moment you ran it — not a dashboard that updates. Re-run update to evolve it in place.

it isn't
A raw transcript dumper

It builds a structured model first and renders that, rather than dumping the verbatim conversation.
