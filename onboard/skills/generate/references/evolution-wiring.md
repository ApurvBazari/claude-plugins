# Evolution Wiring — P8 Notifications → Hook Configuration

This reference documents how `onboard:generate` (v2 path) translates `phases.P8.cicd.notifications` into project-side hook entries during the v2-aware mapping step. The output of this mapping is fed into the `callerExtras.qualityGates.*` shape that the config-generator agent already understands, so the existing `generation/SKILL.md` hook-writing logic handles emission without modification.

This is the "evolution wiring" — the layer that turns CI/CD intent (P8.notifications) into Claude-side automation (settings.json hooks).

---

## What gets wired

| P8 input | Project-side artifact emitted | Why |
|---|---|---|
| `notifications.channels` includes `slack` AND `notifications.events` includes `deploy-failure` | `.claude/hooks/notify-on-deploy-failure.sh` + `Stop` event entry in `settings.json` | Local Slack ping when a deploy job fails. Complements the GHA-side notify step in `deploy.yml.tmpl`. |
| `notifications.channels` includes `slack` AND `events` includes `build-failure` | `.claude/hooks/notify-on-build-failure.sh` + `PostToolUse:Bash` entry matching `npm run build` failures | Catches local build failures too, not just CI failures. |
| `notifications.channels` includes `slack` AND `events` includes `security-alert` | `.claude/hooks/notify-on-security-finding.sh` + entry in `tooling-audit.yml.tmpl` Slack step | Audit findings flagged as security route here. |
| `notifications.channels` includes `discord` (any event) | Same shape as Slack but `DISCORD_WEBHOOK_URL` env var | Discord webhooks have a simpler payload than Slack. |
| `notifications.channels` includes `email` (any event) | `.claude/hooks/notify-on-<event>.sh` using `mailx` or `sendmail` | Solo developer fallback; works without provider-specific webhooks. |
| `notifications.channels` includes only `github-checks` | No project-side hook emitted (GHA status checks are the channel) | Default; pure-GHA flow. |

## Hook script template (Slack)

The generated `.claude/hooks/notify-on-<event>.sh`:

```bash
#!/usr/bin/env bash
# greenfield:notify
# Generated by onboard@2.0.0-alpha.1 from phases.P8.cicd.notifications.
# Edit at your own risk — regenerated by /onboard:evolve.

set -euo pipefail

WEBHOOK_URL="${SLACK_WEBHOOK_URL:-}"
if [ -z "$WEBHOOK_URL" ]; then
  exit 0  # silently skip if secret not configured
fi

EVENT="<event>"                 # substituted at generation
MESSAGE="${1:-no message}"

curl -sS -X POST \
  -H 'Content-Type: application/json' \
  -d "{\"text\":\"[${EVENT}] ${MESSAGE}\"}" \
  "$WEBHOOK_URL" \
  > /dev/null 2>&1 || true       # never fail Claude on notification error
```

ShellCheck-clean. The `|| true` suffix on the `curl` ensures notification failures never block Claude — observability for the developer, never a gate.

## settings.json entries

For `notifications.events.includes("deploy-failure")` with `slack` channel:

```jsonc
{
  "hooks": {
    "Stop": [
      {
        "command": "bash .claude/hooks/notify-on-deploy-failure.sh \"Deploy job ${JOB_NAME} failed\"",
        "matcher": { "errorOccurred": true, "jobName": "deploy-*" }
      }
    ]
  }
}
```

The `matcher` shape mirrors the existing `generation/references/hooks-guide.md § Matcher Compatibility` — onboard 2.x reuses the same matcher vocabulary; only the *event source* (CI/CD vs. wizard) is new.

## Mapping table — caller responsibility

The mapping below is performed inside `onboard:generate` Step 1.5 (v2 path) BEFORE dispatching the config-generator agent. The result is appended to `callerExtras.qualityGates.<event>[]` arrays:

| P8 source | Target `qualityGates` event | Hook config emitted |
|---|---|---|
| `notifications.channels` includes any channel AND `events` includes `build-failure` | `postToolUse` (matching build commands) | `notify-on-build-failure.sh` |
| same channel AND `events` includes `deploy-success` | `stop` (matching deploy jobs) | `notify-on-deploy-success.sh` |
| same channel AND `events` includes `deploy-failure` | `stop` (matching deploy jobs with errorOccurred) | `notify-on-deploy-failure.sh` |
| same channel AND `events` includes `security-alert` | `fileChanged` (matching `*.audit-report.md`) | `notify-on-security-finding.sh` |

For `channels: ["github-checks"]` only (no Slack/Discord/email), no project-side hooks are emitted — GHA's native status checks cover the surface.

## Idempotence

`onboard:evolve` re-runs this mapping when P8 changes. The hook scripts have a `# greenfield:notify` marker comment in their headers (similar to the freshness hook marker); evolve replaces only files carrying that marker. Hand-edited files without the marker are preserved.

---

## Secret variable resolution

The generated hooks read secret values from env vars (`SLACK_WEBHOOK_URL`, `DISCORD_WEBHOOK_URL`, etc.). The variable names are stable across stacks; sourcing them is the developer's responsibility per `phases.P8.cicd.secrets.manager`:

| `secrets.manager` | How developers set the env vars |
|---|---|
| `provider-stored` | Repo secrets configured manually in GitHub UI. CI workflows reference them; local hooks expect the developer to `export` for local invocation. |
| `oidc-to-cloud` | Configured at deploy time only; local hooks have no equivalent (no Slack from local runs unless manually exported). |
| `vault` / `1password` / `doppler` | Developer runs the manager's exec wrapper (e.g., `doppler run -- <command>`) for local sessions. |
| `manual` | Plain `.env` file, sourced before Claude session. |

The hooks do NOT enforce the manager — they exit cleanly when the env var is missing, leaving the choice of source to the developer.

---

## What this reference is NOT

- Not a runtime API. The mapping is performed once at `onboard:generate` time and produces static project artifacts.
- Not extensible at runtime. Adding a new channel (e.g., Microsoft Teams) requires updating this reference + extending the GHA templates. Round 1 ships Slack/Discord/email/github-checks only.
- Not coupled to onboard's evolution hooks. The `notify-on-*.sh` scripts are separate from the drift-detection hooks (`detect-dep-changes.sh` etc.). Both can coexist.

Those additions land in onboard 2.2.0 (Round 3 sync). The Round 1 mapping above is forward-compatible — it doesn't conflict with future P7 enrichment.
