# Harness Design — Long-Running Development Artifacts

Based on Anthropic's "Effective Harnesses for Long-Running Agents" pattern. These artifacts bridge context across Claude sessions, preventing premature completion and maintaining development momentum.

Reference: https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents

## Core Principle

Forge acts as the **Initializer Agent** — it runs once and sets up everything. The artifacts generated here enable every **subsequent Claude session** to quickly understand project state, pick up where the last session left off, and make incremental progress.

The magic isn't in the first session — it's in the artifacts that carry context forward.

## Folder Structure

```
project/
├── init.sh                    ← environment bootstrap (at root for convenience)
├── docs/
│   ├── progress.md            ← cross-session state (updated every session)
│   └── feature-list.json      ← feature tracking (passes: false → true)
├── .claude/
│   └── settings.json          ← SessionStart hook reads docs/progress.md
└── CLAUDE.md                  ← session startup protocol + E2E verification guide
```

State files live in `docs/` (git-tracked, visible, sharable). Preferences and behavioral insights live in Claude Code's memory system (auto-loaded, cross-session). Two layers, two purposes, no duplication.

## Artifact 1: `init.sh` — Environment Bootstrap

Generated at project root. Every Claude session runs this first.

```bash
#!/usr/bin/env bash
set -euo pipefail

# Generated by Forge — run at the start of every Claude session
cd "$(dirname "$0")"

# Install dependencies
[INSTALL_COMMAND]

# Start dev server
[DEV_COMMAND] &
DEV_PID=$!

# Wait for server ready
echo "Waiting for dev server..."
for i in $(seq 1 30); do
  if curl -s http://localhost:[PORT] > /dev/null 2>&1; then
    echo "Dev server running at http://localhost:[PORT] (PID: $DEV_PID)"
    exit 0
  fi
  sleep 1
done

echo "Warning: Dev server did not respond within 30s"
exit 1
```

### Stack-specific commands

| Stack | Install | Dev command | Port |
|---|---|---|---|
| Node.js (npm) | `npm install` | `npm run dev` | 3000 |
| Node.js (pnpm) | `pnpm install` | `pnpm dev` | 3000 |
| Python (pip) | `pip install -r requirements.txt` | `uvicorn app.main:app --reload` | 8000 |
| Python (poetry) | `poetry install` | `poetry run python manage.py runserver` | 8000 |
| Go | `go mod download` | `go run cmd/server/main.go` | 8080 |
| Rust | `cargo build` | `cargo run` | 3000 |
| Ruby | `bundle install` | `bin/rails server` | 3000 |

For CLI tools (no dev server): simpler script that installs deps and runs a smoke test.

## Artifact 2: `docs/feature-list.json` — Feature Tracking

JSON format — models are less likely to inappropriately modify JSON compared to markdown.

```json
{
  "version": "1.0.0",
  "generatedBy": "forge",
  "projectDescription": "[from Phase 1]",
  "sprints": [
    {
      "name": "Sprint 1 — Core Foundation",
      "features": [
        {
          "id": "F001",
          "category": "functional",
          "description": "User can see the main dashboard",
          "steps": [
            "Navigate to /",
            "Verify dashboard layout renders",
            "Verify navigation elements are present"
          ],
          "passes": false,
          "priority": 1
        }
      ]
    }
  ]
}
```

### Generation approach (hybrid)

Feature list is generated during Phase 1 confirmation step. Forge decomposes the app description into testable features, groups them into sprints, and shows the developer:

> **Initial Feature Breakdown** (15 features across 3 sprints):
> [compact list]
> 
> Adjust, add, or skip? (If skipped, a minimal 3-5 feature list is generated.)

### Feature depth by project type

| Project type | Features | Sprints |
|---|---|---|
| CLI tool | 5-10 | 1-2 |
| Simple web app | 15-25 | 2-3 |
| Full-stack production | 30-50 | 4-6 |
| Monorepo | Per-package lists | Varies |

### Critical constraint

Include in every generated feature-list.json and in CLAUDE.md:

> "It is unacceptable to remove or edit feature descriptions or steps.
> You may ONLY change the `passes` field from `false` to `true` after
> verifying the feature works end-to-end."

### Categories

`functional`, `ui`, `data`, `auth`, `integration`, `performance`, `accessibility`

## Artifact 4: Session Startup Protocol (added to CLAUDE.md)

```markdown
## Session Startup Protocol

At the start of every Claude session in this project:

1. **Bootstrap**: Run `bash init.sh` to install deps and start dev server
2. **Orient**: Read `docs/progress.md` for current state and last session's work
3. **History**: Run `git log --oneline -20` to see recent changes
4. **Plan**: Read `docs/feature-list.json` — pick the highest-priority incomplete feature
5. **Verify**: Run baseline tests to confirm existing features still work
6. **Focus**: Work on ONE feature only — don't try to do everything at once
7. **Commit**: When done, commit with descriptive message and update docs/progress.md
```

## Artifact 5: E2E Verification Guidance (added to CLAUDE.md)

```markdown
## Feature Verification

Before marking any feature as `passes: true` in docs/feature-list.json:

1. Test the feature end-to-end as a user would (browser automation preferred)
2. Verify each step listed in the feature's `steps` array
3. If using Playwright MCP: navigate to the page, interact with elements, verify outcomes
4. If no browser automation: manually verify via curl/API calls for backend features
5. Never mark a feature as passing based on code review alone — always test the running app

Do NOT remove or edit feature descriptions or steps. Only change `passes` from false to true.
```

## Two-Layer Persistence Model

```
Git-tracked state (docs/):              Auto-loaded insights (memory):
├── progress.md (session log)           ├── MEMORY.md (index)
├── feature-list.json (what to build)   ├── project_*.md (context)
└── (DISCUSSION.md, PLAN.md — user's    ├── feedback_*.md (workflow prefs)
     own docs, not Forge-generated)     └── reference_*.md (pointers)

PURPOSE: development state              PURPOSE: preferences, knowledge
VISIBILITY: git, CI, other devs         VISIBILITY: Claude sessions only
UPDATED: end of every session           UPDATED: when insights emerge
```

## Single Feature Focus (Critical)

Anthropic found this "critical to addressing the agent's tendency to do too much at once."

Add to CLAUDE.md session startup protocol:

```markdown
6. **Focus**: Work on ONE feature only per session — don't try to do everything at once.
   If you finish early, commit and update progress before starting another.
```

For stronger enforcement, add a SessionStart hook reminder:

```json
{
  "SessionStart": [{
    "hooks": [{
      "type": "prompt",
      "prompt": "Remind the developer to focus on ONE feature at a time from docs/feature-list.json. Read the feature list and suggest which feature to work on next."
    }]
  }]
}
```

## Test Immutability

From Anthropic: "It is unacceptable to remove or edit tests because this could lead to missing or buggy functionality."

Claude commonly rewrites tests to make them pass instead of fixing the underlying code. Add to CLAUDE.md:

```markdown
## Test Immutability Rule

Do NOT modify existing tests to make them pass. If a test fails, fix the implementation, not the test.
Tests are immutable once written — they define the expected behavior.

The only acceptable test changes are:
- Adding NEW tests for new features
- Fixing a test that was genuinely wrong (wrong expected value, testing the wrong thing)
  — but this requires explaining WHY the test was wrong, not just that it failed
```

## Context Anxiety Mitigation

Sonnet models may prematurely wrap up work near perceived context limits — rushing to finish instead of maintaining quality. Add to CLAUDE.md:

```markdown
## Context Management

If you notice the conversation is getting long:
- Do NOT rush to finish the current feature. Quality matters more than completion.
- Commit your current progress with a descriptive message.
- Update docs/progress.md with what's done and what's remaining.
- It's better to end a session with a well-committed partial feature than a rushed complete one.
- The next session will pick up exactly where you left off via the progress file.
```

For developers using Sonnet for teammates: the agent teams guide should recommend context resets (clearing and re-spawning teammates) for sprints with many features.

## Verification Reports (File-Based Communication)

Anthropic uses files for inter-agent communication: "one agent would write a file, another would read it."

The evaluator writes reports to `docs/verification-reports/`:
```
docs/
├── verification-reports/
│   ├── sprint-1-2026-04-05.md      ← evaluator writes
│   ├── sprint-1-2026-04-08.md      ← second run after fixes
│   └── feature-F003-2026-04-07.md  ← single feature check
```

These create an auditable trail. Previous reports can be compared to determine whether the REFINE or PIVOT recommendation makes sense.

## Cost Tracking

Add to `forge-meta.json` and `docs/progress.md`:

```json
// forge-meta.json
{
  "costs": {
    "forgeInit": {
      "duration": "25 minutes",
      "estimatedTokens": "~150K",
      "phases": {
        "contextGathering": "~30K",
        "scaffold": "~20K",
        "toolingGeneration": "~100K"
      }
    }
  }
}
```

Add to `docs/progress.md` session log:
```markdown
### Session 2 — [date]
- Implemented F001 (user dashboard)
- Duration: ~45 minutes
- Tokens: ~80K (estimated)
```

Add to CLAUDE.md:
```markdown
## Cost Awareness

Each Claude session uses tokens. Rough estimates:
- Simple feature implementation: ~50-100K tokens (~$3-8)
- Complex feature with debugging: ~100-200K tokens (~$8-15)
- /forge:verify run: ~20-80K tokens (~$1-8)
- Sprint verification: ~50-150K tokens (~$3-12)

To minimize costs: focus on one feature per session, commit frequently, use the progress file to avoid re-orientation.
```

## Stress-Testing Harness Assumptions

From Anthropic: "Every component in a harness encodes an assumption about what the model can't do on its own, and those assumptions are worth stress testing."

### Which Components Are Assumptions

| Component | Assumption | Test by removing |
|---|---|---|
| Feature list | Claude won't track what's done/remaining without a list | Try without: does Claude attempt too much or forget features? |
| Sprint contracts | Claude won't maintain quality standards without explicit criteria | Try without: does quality degrade? |
| Evaluator agent | Claude can't honestly evaluate its own work | Try self-evaluation: does it inflate results? |
| Single-feature focus | Claude will try to do too much at once | Try multi-feature: does quality drop? |
| Worktree isolation | Claude will accidentally break main | Try direct-branch: do conflicts/accidents increase? |
| init.sh | Claude will waste time figuring out how to run the app | Try without: how long does orientation take? |
| Progress file | Claude can't maintain context across sessions without a file | Try without: does it repeat work or miss context? |

### When to Re-Evaluate

- After a major model upgrade (e.g., new Sonnet or Opus release)
- After completing 3+ sprints on a project (real-world data)
- When a component feels like overhead rather than help

### How to Test

Remove one component at a time and observe impact over 2-3 sessions. If removing it doesn't degrade quality, simplify the harness. Models improve fast — what was essential for Sonnet 4.5 may be unnecessary for Opus 4.6.

## When to Generate

All harness artifacts are generated during Phase 3 (AI Tooling), after scaffold is complete:

1. `init.sh` — written to project root, made executable
2. `docs/feature-list.json` — written from Phase 1 feature decomposition
3. `docs/progress.md` — initialized with Session 1 (Forge init) entry
4. `docs/verification-reports/` — directory created (empty until first /forge:verify run)
5. Session startup protocol — added to CLAUDE.md via onboard headless `callerExtras`
6. E2E verification guidance — added to CLAUDE.md
7. Single feature focus rule — added to CLAUDE.md
8. Test immutability rule — added to CLAUDE.md
9. Context anxiety mitigation — added to CLAUDE.md
10. Cost awareness section — added to CLAUDE.md
