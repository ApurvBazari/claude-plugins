Commands & Gate Rigor

Every harness run is driven by a command. The command determines which phases run and what gate rigor applies at each phase — from zero-friction patch to full-rigor feature.

feature
Full SDLC rigor. Hard gates on spec, plan, verify, review.
fix
Speed-optimized. Diagnose fast, mostly auto gates.
refactor
Behavior-must-not-change. Hard on plan, verify, review.
spike
Exploration only. All advisory — all soft gates.
patch
Zero friction. All auto, research phase skipped.
optimize
Benchmark-focused. Hard gate on verify only.
docs
Readability-focused. Mostly soft gates throughout.
security
Audit & remediation. Hard on plan, verify, review.
integrate
API/service wiring. Hard on spec, plan, verify, review.
structure
Project reorganization. Hard on plan and verify.

Micro-Cycles & Escalation

Every phase runs in a tight retry loop before escalating. When retries are exhausted, the escalation table determines how far back to rewind the pipeline.

Phase micro-cycle loop
ATTEMPT
Execute phase work
→
CHECK
Run gate checks
→
pass
ADVANCE
Transition to next phase
FAIL
Gate not passed
→
DIAGNOSE
Identify root cause
→
FIX
Apply targeted fix
↺ After FIX, loop back to ATTEMPT. Max 3 retries per phase before escalation is triggered.

Escalation Severity Table triggered after 3 failed retries

Severity Examples Action Phases Replayed
Minor Lint failures, coverage slightly under threshold, formatting Stay in current phase, flag for manual review none — current phase retried
Moderate Test failures, type errors, integration breaks after implementation Escalate to BUILD phase P3 → P4 → P5
Major Architecture conflicts, missed dependencies, wrong abstraction Escalate to PLAN phase P2 → P3 → P4 → P5
Critical Wrong problem solved, acceptance criteria violated, security vulnerability in design Escalate to SPEC phase, human review mandatory P1 → P2 → P3 → P4 → P5

Cascade Rule

Escalating to an earlier phase doesn't mean starting from scratch — it means re-running from that phase forward through the full gate sequence.

DX Modes

The harness ships three developer experience modes. Set globally in config, override per-session with a flag, or let the command default determine it.

⚡ Flow mode quiet

Silent, non-interruptive. Only surfaces critical failures that block progress. Everything else runs in the background.
✓ Background gate checks — no interruption
✓ Auto-advance on passing gates
✓ Surfaces only critical hard-gate failures
○ Soft gate warnings batched at phase end
○ Best for: patch, fix, docs

🔖 Checkpoint mode default

One prompt per phase transition. Bold default makes it fast. Optionally drill into details or pause at any point.
✓ Single prompt per gate: [Advance] / View details / Hold
✓ Bold default means Enter = advance
✓ All gate results visible in summary
○ "Hold" pauses pipeline for intervention
○ Best for: feature, refactor, integrate

🧭 Guided mode verbose

Active suggestions and step-by-step narration. Claude walks you through each decision point.
✓ Explains each phase before starting
✓ Suggests next action at every step
✓ Full gate detail at each transition
○ Higher ceremony budget usage
○ Best for: onboarding, security, complex spikes

The harness tracks per-feature state across sessions and supports multiple parallel features via git worktrees — each in its own phase.

Override Stack

highest

Layer 3: Session Override

One-off flags: --skip-mutation, --gate=soft, --phase=build, --dx=flow. Ephemeral — never persisted to state.

project

Layer 2: Project Config

harness.config.json in project root. Overrides profile defaults for this project. Set worktrees directory, DX mode, phase thresholds, render config.

default

Layer 1: Profile Preset

Ships with the harness plugin. prototype, solo-dev, or production. Provides sensible defaults that project config can selectively override.

Example: harness.config.json (v2)
