Audit of every project concern a real greenfield session should cover, grouped by phase, with a coverage badge per concern. Generated 2026-05-12 in response to: "time is not a constraint, surprises are the enemy — the primary job of this plugin is to get the best output without any surprises."

Headline numbers

Current wizard: 33 questions across 7 categories. Developer answers 8–22 per session.
Identified gaps below: ~80 missing questions across the same categories plus ~12 missing concern areas never asked about at all.

Section 3 — Missing phases (beyond questions)

These are entire interactive phases / sub-skills that don't exist today.

Missing phaseWhat it does
Domain ModelingExtract core entities and their relationships ("User has many Posts; Post belongs to User; Comment belongs to Post and User"). This becomes the foundation for DB schema, API design, ER diagrams. Currently emerges implicitly from features.
Persona & User JourneysIdentify 2–3 primary user personas, their goals, pain points, and key journeys through the system. Drives feature priority and verification scenarios.
Risk IdentificationTechnical risks (scale, complexity, novel tech), business risks (market, regulation, competition), security risks (attack surface, threat model). Informs test strategy, monitoring, deployment caution.
Feature Roadmap Co-DesignCurrently docs/feature-list.json is auto-emitted. Should be an interactive phase where dev confirms feature boundaries, dependencies, MVP vs deferred, sprint sizing.
Schema / API Draft ReviewBefore scaffolding writes any DB code, present a proposed schema (tables, columns, indexes) and a proposed API contract (routes, request/response shapes). Get explicit approval. This is where most cascading misalignments would surface.

Section 4 — Proposed new wizard structure

Replaces the current 7 categories with 10 phases. Each phase has its own confirmation/synthesis checkpoint (per the Phase 1.8 pattern already agreed). Phases are heavily adaptive — most projects answer 30–60 questions out of ~150 possible.

Each phase ends with a synthesis confirmation (the Phase 1.8 pattern previously agreed): an HTML page summarizes what was captured, dev clicks Approve / Adjust / Skip per section before moving to the next phase. This way misalignment is caught at phase boundaries, not at the end.

Why this isn't 1× big synthesis review at the end

By the time you reach Phase 8 (CI/CD), you've already made decisions in Phase 3 (DB) that depend on Phase 6 (auth model). A single synthesis review at the end would require either: (a) walking everything top-down again (slow), or (b) limited backtracking ability if a late decision invalidates an earlier one. Per-phase confirmation catches contradictions immediately, when the relevant context is fresh.

Section 5 — Implementation phasing

Given the scope, this can't be one branch / one PR. Proposed sequencing:

RoundScopeWhy this order
Round 1 Expand Cat 5 (CI/CD) and add Phase 1.8 Synthesis Review for the synthesis pattern itself The user explicitly flagged CI/CD. Building Phase 1.8 establishes the synthesis-confirmation infrastructure that subsequent rounds will reuse.
Round 2 Expand Cat 3 → split into Data Architecture (P3) and API & Integration (P4); add per-phase synthesis Second-largest gap area. Data + API decisions cascade into auth, validation, frontend.
Round 3 Expand Cat 4 → Workflow & Quality (P7); add Auth, Security & Privacy (P6) split-out Workflow + security have similar light-asking patterns. Best done together to keep the question-bank refactor coherent.
Round 4 New phases: Personas (P0.5), Domain Modeling (P1), Risk ID (P8.5) These are new infrastructure rather than question expansions — needs new SKILL files, references, and (probably) new agents.
Round 5 Feature Roadmap Co-Design (P9), Schema & API Draft Review (P10.5) These integrate with Phase 2 scaffolding. Best done after the upstream phases are stable so cascading changes don't reverberate.
Round 6 Frontend & UX expansion (P5); concern areas never asked about (caching, search, real-time, payments, etc.) Lower-priority for typical projects; many are conditional. Done last so the wizard's conditional-skip logic can be exercised against a mature base.

Section 6 — Open design questions for the user

Listed here for tracking; will be discussed in the chat following this doc.

  • Do we want all phases to end in a synthesis confirmation, or only "major decision" phases (P3, P6, P8, P10.5)?
  • Should the synthesis HTML files be committed to the project repo? (Pro: living architecture record. Con: noise / re-gen churn.)
  • For new phases that need agents (Risk Identification might want a research agent), do we add them to greenfield/agents/ or compose existing ones?
  • Do we want a "minimal" mode that turns off the comprehensive wizard for users who just want to ship a side project? Or is the new default always comprehensive?
  • How do we handle phase resume? If a session is interrupted at P6 Q5, does /greenfield:resume restart P6 from scratch or continue mid-phase?
  • For "concern areas never asked about" (caching, real-time, payments), should the wizard ALWAYS ask "do you need this?" first, or only ask when stack/scale signals make it likely?
