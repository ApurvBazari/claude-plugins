What is greenfield?

Greenfield is a Claude Code plugin that scaffolds a new project from a single command — /greenfield:start — and walks you through an adaptive wizard that captures every architectural decision you'd otherwise leave implicit. By the time the scaffold runs, you've already signed off on topology, data, API, auth, security, CI/CD, and feature decomposition. Each decision becomes a versioned ADR. The plugin then delegates to onboard for Claude tooling (CLAUDE.md, rules, skills, agents, hooks) and to a curated catalog of plugins for the rest of your workflow. You end up with a project that is AI-native from day one.

The running example: Nimbus Notes

Nimbus Notes — a multi-tenant SaaS for AI-summarized meeting notes.

Stack: Next.js 15 (App Router) · Postgres 16 + Prisma · Clerk auth · Stripe billing · deployed on Vercel · GitHub Actions CI · scale target: hobby → startup. This example threads through every phase below so you can see what greenfield asks, what it generates, and how the pieces connect.

The high-level flow

From the developer's POV, /greenfield:start takes you through 3 super-phases. Behind the scenes, those are 8 sub-phases today (15 once Round 3 ships) plus delegation to onboard and a curated plugin catalog.

The forward roadmap

Greenfield ships in rounds on feat/greenfield-1.2. Each round expands the wizard with new phases. Future rounds reuse the Round 2.5 foundation (stale-flags, synthesis reviews, adjust-dialog, MD/HTML drift hooks) — they're content authoring on top of locked infrastructure.

Where this fits in the ecosystem

Greenfield is one of four sibling plugins. It hands off to onboard for the heavy tooling generation, and recommends a set of other plugins (superpowers, commit-commands, hookify, etc.) to install into the new project. The boundary is deliberate: greenfield knows about creating projects; onboard knows about analyzing projects and producing tooling; the other plugins handle ongoing workflow.

Why this matters when you're starting a project

Most project scaffolders generate code. Greenfield generates decisions. Eight key wins, each with a concrete Nimbus Notes example showing what would otherwise go undocumented or wrong.

1. Decisions become ADRs by default

Every architectural choice flows through a synthesis review that writes both a styled HTML and a markdown ADR to docs/adr/ — version-controlled, diff-able, and reviewable in PRs.

Nimbus example: "Why modular-monolith over microservices?" is answered in docs/adr/architectural-framing.html with the constraint that drove it ("solo dev + startup scale"). Future-you doesn't have to remember.

2. Per-phase synthesis review prevents drift

Each major phase ends with an Approve/Adjust/Skip walk. "Adjust" triggers the 5-category adversarial dialog (Scope, Assumptions, Alternatives, Risks, Dependencies). You can't accidentally absorb a half-formed decision.

Nimbus example: Wanted to "just pick Mongo" mid-wizard. Adjust dialog asked "what's the multi-tenancy story?" and "does Prisma support Mongo today?" — caught the impedance mismatch before scaffolding.

3. Stale-flag propagation auto-detects re-walks

When you adjust an upstream phase, every downstream dependent gets marked stale. On entry to a stale phase the wizard offers a re-walk — no more "I changed X but forgot Y still assumed the old value."

Nimbus example: Switched cache from "none" to "redis" after API step. apiIntegration → rate-limit went stale; re-walk surfaced that Upstash Ratelimit was now the obvious pick over an in-memory map.

4. Walking-skeleton mode for complex stacks

For multi-layer architectures, instead of generating a full app you can generate one-of-each: one entity, one route, one service, one test. Phase 3 then learns from real code; Phase 2b expands under AI-tooling guidance.

Nimbus example: Not used here (Next.js + Prisma is well-understood), but for an Android Compose + Hilt + Room project, a walking skeleton dramatically improves the quality of generated rules and skills because they're grounded in code.

5. Cross-plugin reuse — greenfield doesn't reimplement

Phase 3b delegates the heavy lifting (CLAUDE.md, rules, skills, agents, hooks, CI/CD workflows, sprint contracts) to onboard's headless generate skill. Greenfield knows about new projects; onboard knows about analyzing them. Two plugins, clear boundary.

Nimbus example: onboard generated Prisma-specific rules (.claude/rules/prisma.md) and tRPC procedure conventions because greenfield passed the stack analysis verbatim. Greenfield never had to know about Prisma idioms.

6. Resume-anywhere via atomic state checkpoints

Every Step boundary writes .claude/greenfield-state.json atomically (.tmp + rename). Ctrl+C any time, come back days later — /greenfield:pickup reads the state and resumes at the exact step you were on. Mid-synthesis-review? Mid-feature-decomposition? Works.

Nimbus example: Started wizard Wednesday morning, got pulled into a meeting at Step 4. Came back Thursday — /greenfield:pickup showed completed steps, dropped me right back at the API question I was on.

7. Stack-derived defaults beat blank-form-fatigue

Every wizard question shows a [default: X] derived from your stack + earlier answers. Press Enter to accept; type to override. Round 3 spec calls for ~50 explicit Qs becoming ~30-40 with defaults. Wizard is comprehensive without being exhausting.

Nimbus example: "Next.js + multi-tenant ⇒ default auth: hosted · clerk." Accepted with Enter. "TypeScript + strict ⇒ default lint: eslint-strict + prettier." Accepted. The wizard mostly confirmed sensible choices instead of asking me to invent them.

8. Adversarial 5-category adjust-dialog

When you pick "Adjust" on any synthesis section, the dialog walks Scope → Assumptions → Alternatives → Risks → Dependencies. One probe per category, max one follow-up. You can't change a decision casually — you have to argue past 5 buckets of friction first.

Nimbus example: Adjusted CI/CD coverage from 80% to 70%. Dialog asked: "Risks: what code paths most need coverage but might not get it at 70%?" — surfaced that billing webhooks (F010) needed a hard-coded gate even if global threshold dropped.

9. Sprint contracts & independent verification

Step 8 (Feature Decomposition) produces docs/feature-list.json with verification steps per feature. onboard generates sprint-contracts/sprint-N.json with mustPass[] criteria. /onboard:verify F001 runs the feature in a worktree and judges objectively — no marking your own homework.

Nimbus example: F002 (paste transcript page) had a verification step "Edge case: empty input shows validation error". /onboard:verify F002 opened a browser, hit the page, submitted empty, confirmed the error appeared. PR could merge with confidence.

When greenfield is NOT the right tool

Worth being honest about the boundaries:

  • Existing projects. Use /onboard:start instead — it analyzes what's already there and produces tooling for the existing codebase. Greenfield assumes an empty directory.
  • Quick prototypes / throwaway code. The wizard is comprehensive on purpose. For a 30-min hackathon scratchpad, the overhead isn't worth it.
  • Stacks the catalog doesn't know about. Defaults degrade gracefully (stack-researcher does live web research), but a Roc/Gleam/Crystal project will feel less guided than Next.js/Django/Rails.
  • When you have strong opinions already documented elsewhere. Greenfield is most useful when the architectural decisions are unmade. If you have a battle-tested template repo, Path C (template scaffold) lets you skip the wizard entirely.

Why this gets better over time

Every round adds new phases on top of the Round 2.5 foundation (stale-flags, adjust-dialog, MD/HTML drift hook, synthesis templates). The phases are pure content authoring — no infrastructure work. So Round 3 ships ~40 files, Round 4 maybe similar, Round 6 similar.

By Round 6, the wizard will have asked you about 22+ named phases covering vision, personas, domain model, architecture, data, API, auth, privacy, security, runtime ops, frontend, CI/CD, features, risks, roadmap, schema preview — every cross-cutting concern that usually surfaces as a bug in week 3.

The strength compounds: more phases mean more synthesis reviews mean more cross-phase dependency checks mean less drift between intent and implementation. The goal isn't a faster scaffold. It's a scaffold whose decisions you've actually made and can defend.
