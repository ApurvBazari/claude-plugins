The End-to-End Flow

0 Entry — /greenfield:start Skill (user-invoked)
User runs /greenfield:start. Greenfield checks .claude/greenfield-state.json — if a prior session is in flight, offers /greenfield:pickup. Otherwise, starts fresh and seeds the state file.
▼
1 Context Gathering — adaptive wizard 11 steps
State-machine wizard runs ~30–57 questions (out of ~75 possible) based on prior answers.
  • Step 1 App description & type (Project Vision)
  • Step 2 Tech stack → dispatches stack-researcher agent for web research (latest versions, scaffold CLI, best practices)
  • Step 2.5 Architectural Framing — topology, deployment shape, scale target, boundary notes (4 Qs; synthesis fires inline)
  • Step 3 Data Architecture — DB engine + host, ORM, migrations, multi-tenancy, caching, file storage, codegen, backup, compliance (12 Qs; synthesis fires inline)
  • Step 4 API & Integration — style, versioning, rate limits, async patterns, real-time, webhooks, external services (10 Qs; synthesis fires inline)
  • Steps 5–9 Remaining project details, pain points, workflow preferences, CI/CD & Delivery (Step 7 synthesis), feature decomposition, confirmation
  • Step 10 Architectural Research (conditional — only if parked questions exist)
  • Step 11 Architectural Validation — cross-phase sign-off, divergence table, contradiction check (1–2 Qs; synthesis fires inline)
Output: a populated context object inside greenfield-state.json — drives every downstream decision.
▼
1.5 Architectural Research (conditional) Skipped if no parked questions
If the wizard "parked" deep-research questions ("Which on-device LLM tier?", "Multi-tenant DB structure?"), this phase resolves them with focused web/codebase research, then yields back to grilling.
▼
1.8 Synthesis Review — per-phase confirmation gate (NEW in 3.0) Inline sub-step
Runs inline at the end of each major wizard step that has a synthesis template. After Round 2.5 (prep for Round 3), this is wired for 9 phases: Step 2.5 (Architectural Framing / architecturalFraming), Step 3 (Data Architecture / dataArchitecture), Step 4 (API & Integration / apiIntegration), Step 5.1 (Auth / auth), Step 5.2 (Privacy / privacy), Step 5.3 (Security / security), Step 5.4 (Runtime Operations / runtimeOperations), Step 7 (CI/CD & Delivery / cicdAndDelivery), and Step 11 (Architectural Validation / architecturalValidation) — later rounds add per-phase synthesis for workflow, vision/personas/domain, featureRoadmap, and schemaDraftReview.
  • Renders docs/architecture/<topic-kebab>.html in the scaffolded project — a living architecture record, ADR-style, committed to the repo. (e.g. architectural-framing.html, data-architecture.html, api-integration.html, cicd-and-delivery.html, architectural-validation.html)
  • Walks the developer through each section: Approve / Adjust / Skip. Adjust composes superpowers:brainstorming + mattpocock-skills:grill-me (or falls back inline).
  • Writes a machine-readable <topic-kebab>-dependencies.json sidecar for cross-phase dependency tracking. scripts/visualize-graph.sh renders the whole graph as Mermaid on demand.
  • Installs a one-time pre-commit hook (warning-only) that flags architectural changes lacking a docs/architecture update.
  • Brief currentPhase transition: phase-1-context-gathering → phase-1.8-synthesis-review → phase-1-context-gathering. currentSynthesisPhase records which phaseId is in progress.
▼
1.7 Grill-Spec — pre-scaffold validation gate Plugin point
Walks the spec one more time and surfaces contradictions before any scaffolding starts. Five categories: Scope, Stack alignment, Feature conflicts, Missing dependencies, Security/compliance.
Plugin used: mattpocock-skills:grill-me (vendored in this repo). The skill invokes it via the Skill tool. If unavailable for any reason, falls back to an inline 5-category walk — gate works either way.
▼
2 Scaffolding Two modes
Executes the actual project scaffold based on Step 3's choice:
  • full: runs the official CLI (create-next-app, uv init, cargo new, etc.) with research-informed flags
  • walking-skeleton: produces one of each architectural pattern (one entity, one DAO, one route, one test) so Phase 3's tooling can learn from real code
Initializes git, verifies "hello world" runs, writes initial commit.
▼
3a Plugin Discovery — install & verify ★ MAIN plugin point
Matches a curated catalog (greenfield/skills/plugin-discovery/references/plugin-catalog.md) against the Phase 1 context.
  • Step 1 Match catalog rules → ranked list
  • Step 2 Present interactive checklist (multi-select), phase-grouped (Research → Core discipline → Per-feature → Review → Engineering → Guardrails)
  • Step 3 Optional: web-search for extras beyond the catalog
  • Step 4 Install via shell CLI: claude plugin marketplace add <src> + claude plugin install <name>@<mkt> --scope user
  • Step 4b Verify via claude plugin list --json after user runs /reload-plugins
  • Step 5 Compile coveredCapabilities[] from verified plugins
Plugins typically installed here: superpowers, commit-commands, security-guidance, hookify, claude-md-management, notify, plus stack-conditional ones (frontend-design, playwright, context7) and the vendored mattpocock-skills.
▼
3b Tooling Generation — delegated to onboard:generate ★ Plugins INFLUENCE output
Calls onboard:generate headlessly with callerExtras = { installedPlugins, coveredCapabilities }. Onboard produces the project's Claude tooling:
  • CLAUDE.md — including a Plugin Integration section listing installed plugins
  • .claude/rules/*.md — project conventions referencing plugin slash commands
  • .claude/skills/*, .claude/agents/*.md — only gap-filling; capabilities covered by plugins are skipped
  • .claude/hooks/*.sh — standard 5 hooks (per project-standards.md)
  • .github/workflows/*.yml — CI/CD pipelines
  • docs/sprint-contracts/*.json — sprint exit criteria
Greenfield itself adds two stack-specific artifacts: init.sh and docs/feature-list.json.
Shadowing prevention: if coveredCapabilities contains "code-review" (from code-review plugin), onboard skips generating a local code-reviewer.md agent. Without this, you'd end up with two competing reviewers.
▼
4 Handoff Summary + next steps
Prints a summary of what was scaffolded + installed, the path to start working (./init.sh, docs/feature-list.json), and suggested next commands. State file marked completedAt.

Key Questions Answered

Your question Answer
Are the installed plugins' contents copied into the generated CLAUDE.md? No — their content lives in ~/.claude/plugins/cache/<marketplace>/<plugin>/<version>/. The generated CLAUDE.md only references them (a "Plugin Integration" section listing names + capabilities), and surrounding rules/skills point at plugin slash commands like /superpowers:tdd. Plugins remain a separate concern, loaded by Claude Code's plugin system.
Does CLAUDE.md tell future sessions "these plugins are the base for feature/fix work"? Yes, indirectly. The "Plugin Integration" section + workflow rules say things like "Use /feature-dev:feature-dev for new feature work" and "Run TDD via /superpowers:tdd". Future sessions in the same project see this CLAUDE.md, so the LLM knows the canonical workflow expects these plugins.
If a developer doesn't have a plugin installed, does it break? Partially graceful. Plugins install at --scope user, so they persist for ANY project the developer opens. Greenfield assumes the same user; if a teammate clones the repo, they'd need to re-install. There's no enforcement — CLAUDE.md mentions them but Claude Code doesn't error if absent. (This is a known gap — see Discussion Log item #1.)
What happens if I install a NEW plugin later (post-scaffold)? Two paths today: (1) Manually edit CLAUDE.md to mention it, or (2) run /onboard:evolve which re-runs tooling generation with the updated plugin list, refreshing the Plugin Integration section. Hooks may also queue a drift entry (.claude/drift-queue.json) prompting you to evolve.
What if upstream plugin authors push breaking changes? claude plugin update <name> pulls latest; the CLI notes "restart required to apply". For the vendored mattpocock-skills, our scripts/sync-from-upstream.sh diffs against upstream HEAD and reports drift — pulling updates is manual + version-bump-tracked.
Are plugin-provided skills auto-invokable, or do I have to slash-command them? Per-skill: each SKILL.md has frontmatter (user-invocable, disable-model-invocation) that controls this. Most skills are model-auto-invokable when their description matches the task — which is why description quality matters more than name.

What's "Baked In" vs "Loaded Dynamically"

Baked into the project repo

  • CLAUDE.md (root + per-area)
  • .claude/rules/*.md
  • .claude/skills/* (project-specific only)
  • .claude/agents/*.md (project-specific only)
  • .claude/hooks/*.sh
  • .github/workflows/*.yml
  • docs/feature-list.json
  • docs/sprint-contracts/*.json
  • init.sh

Loaded dynamically from plugins

  • Plugin SKILL.md files (from ~/.claude/plugins/cache/)
  • Plugin agents/*.md
  • Plugin hooks/*.sh
  • Plugin commands/* (slash commands)
  • Plugin MCP servers (defined in .mcp.json within the plugin)

Claude Code's plugin system auto-discovers all of these at session start (or after /reload-plugins). They're available across all projects opened by the same user.

Plugin Integration: The 5 Touchpoints

# Where What it does with plugins Notes
1 Phase 1.7 grill-spec Invokes mattpocock-skills:grill-me via Skill tool if available Has inline fallback — gate works without the plugin
2 Phase 3a Step 1 catalog match Reads plugin-catalog.md, matches conditions vs context The catalog is curated; web-search adds non-catalog plugins via Step 3
3 Phase 3a Step 4 install Runs claude plugin marketplace add + claude plugin install per selected entry --scope user default. Per-plugin failures non-fatal.
4 Phase 3a Step 4b verify Parses claude plugin list --json after user reloads, asserts enabled === true Unverified plugins get verified: false and don't contribute to capabilities
5 Phase 3b onboard generation Passes installedPlugins[] + coveredCapabilities[] via callerExtras Onboard skips generating agents that would shadow plugins; CLAUDE.md gets a Plugin Integration section

Important: plugins are user-scoped, not project-scoped

The default --scope user install means a single set of plugins exists per developer machine, used across every Claude Code project they open. A project's CLAUDE.md tells the LLM "expect these to be present," but doesn't enforce it. A teammate cloning the repo would need to re-install the same plugin set manually. There's no package.json-style lockfile for plugin dependencies (yet).

  • P3 Data Architecture — 12 questions covering DB engine + host, ORM, migrations, multi-tenancy, search, caching, file storage, codegen, backup, compliance. 7 synthesis sections, 4 required enum-locked schema fields (databaseHost, orm, migrationsTool, multiTenancy).

Locked design decisions for Round 2:
  1. Single-owner boundary — file storage + codegen + caching live in P3 only; real-time + webhooks + jobs live in P4 only. Synthesis HTMLs cross-reference each other when needed.
  2. Hybrid schema strictness — 3–5 required enum-locked fields per phase (the ones other phases cross-reference); remaining fields are loose strings with "other" escape hatches. Avoids monthly schema releases as new ORMs / DB hosts emerge.

2026-05-13 · ROUND 2.5 LOCKED · Foundation pass — 8 cross-cutting decisions before Round 3
A post-Round-2 honest-assessment session (“will devs use this? what's broken structurally?”) surfaced 8 cross-cutting decisions that affect every remaining round. Locking them before Round 3 starts on P6/P7 prevents drift and avoids a much larger Round 6 cleanup. The consolidated retrofit work is scoped as Round 2.5 “Foundation pass” (~50-70 files).

The 8 decisions:
  1. Lite mode / on-ramp: defaults-driven skip. Every Q displays a smart default; Enter accepts, type to override. Synthesis HTMLs still produced. Adjust loop still offered. Preserves the “deliberation is the thesis” position while letting fast devs blast through.
  2. Defaults source: stack-derived from earlier answers; greenfield-opinionated fallback when no stack signal exists. Every Q in question-bank.md gets a default: derivation rule.
  3. Synthesis format: hybrid — independently authored markdown details + HTML executive summary per phase. PR reviewers get clean diffs; browser readers get richness. 2× authoring cost accepted; drift-check hook added between the two files.
  4. Approval semantics / late Adjust: stale-flag with explicit re-walk choice. When P3 is adjusted, downstream phases that reference changed fields are marked stale; on next entry the wizard prompts “P8 references P3.databaseHost which changed — re-walk P8? Y/n”. Requires dependency-graph traversal + field-level diff capture.
  5. Naming consistency: topic names everywhere. Drop P-codes from schema, code, filenames, dependency refs. p3Data → dataArchitecture; p3-data.html → data-architecture.html. User-facing prompts still say “Step 3: Data Architecture”. Eliminates the current 5-conventions-for-one-concept cost.
  6. Architectural Research positioning: split into two passes. Early Architectural Framing (new Step 2.5, 3-5 Qs: monolith/microservices/modular, deployment topology, scale target) informs P3-P9. Late Architectural Validation (new Step 11, always-run final cross-phase sign-off) catches gaps the framing missed. Existing Step 10 (Architectural Research, conditional parked-question resolution) is preserved as a separate concern. Wizard count 10 → 11.
  7. Adjust dialog / mattpocock-skills: build greenfield-owned adjust-dialog skill (5-category adversarial walk: scope / assumptions / alternatives / risks / dependencies). Remove mattpocock-skills from the marketplace entirely. Greenfield owns its core thesis end-to-end; no external plugin dependency for Adjust. The 4-plugin marketplace becomes 3 (or 4 once handoff merges).
  8. Synthesis directory naming: docs/adr/ (was docs/architecture/) in scaffolded projects. Industry-standard ADR convention; instantly recognizable.
Plus state-JSON evolution policy: hard cutover during alpha; migrations from 3.0.0 stable onward. In-flight session breakage documented in CHANGELOG per alpha bump.
