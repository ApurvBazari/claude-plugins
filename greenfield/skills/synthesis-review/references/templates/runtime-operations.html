1. Background Jobs & Retries
jobs.provider:                    {{runtimeOperations.jobs.provider}}

retryStrategy.deliveryGuarantee:  {{runtimeOperations.retryStrategy.deliveryGuarantee}}
retryStrategy.retryPolicy:        {{runtimeOperations.retryStrategy.retryPolicy}}
retryStrategy.deadLetterQueue:    {{runtimeOperations.retryStrategy.deadLetterQueue}}
Gated on apiIntegration.asyncPattern = {{apiIntegration.asyncPattern}}: this section is rendered only when async pattern is not none. At-least-once delivery (deliveryGuarantee) is the default guarantee for most queue-based systems (SQS, BullMQ, Inngest) — it requires consumers to be idempotent (processing the same job twice must be safe). At-most-once delivery is simpler but risks job loss on worker crash. Exactly-once delivery is expensive and rarely worth the complexity for most workloads. retryStrategy.retryPolicy should include backoff strategy (exponential recommended) and a maximum attempt count. retryStrategy.deadLetterQueue is mandatory for production-scale: failed jobs that exhaust retries must be routed to a dead-letter queue for inspection and replaying rather than silently dropped. Platform note: architecturalFraming.deploymentShape = {{architecturalFraming.deploymentShape}} influences the job provider default (Inngest for serverless/Vercel, BullMQ/Redis for container-based, Sidekiq for Rails, Celery for Python).
2. Scheduled Tasks
scheduling.tool: {{runtimeOperations.scheduling.tool}}
Platform default derives from architecturalFraming.deploymentShape = {{architecturalFraming.deploymentShape}}: Vercel Cron for Vercel deployments, GitHub Actions cron for CI-driven or GitHub-hosted projects, Kubernetes CronJob for k8s clusters, Inngest or Temporal for orchestration-heavy workloads. Scheduling tool choice also affects observability — platform-native schedulers (Vercel Cron, GH Actions) surface execution history in the platform UI; third-party schedulers (Inngest, Temporal) provide richer retry, replay, and observability out of the box. For stateful scheduled tasks (e.g., database cleanup, report generation), ensure the scheduled runner has access to the correct environment credentials and that idempotency is enforced (Section 1 delivery policy applies here as well).
3. Observability — Metrics
metrics.tool: {{runtimeOperations.metrics.tool}}
A metrics tool (Datadog, Prometheus + Grafana, AWS CloudWatch, Honeycomb, New Relic) is the primary signal for capacity planning, SLO tracking (Section 7), and alerting (Section 6). Key metric categories for most applications: request rate, error rate, and latency (RED method); CPU, memory, and disk utilisation (USE method); business metrics (jobs processed per minute, active users, payment conversion rate). Prometheus + Grafana is the open-source standard for containerised workloads; Datadog and New Relic are managed SaaS options with lower operational overhead; Honeycomb is the preferred choice when traces (Section 4) are the primary observability signal.
4. Observability — Traces
traces.otel:    {{runtimeOperations.traces.otel}}
traces.backend: {{runtimeOperations.traces.backend}}
Gated on architecturalFraming.topology = {{architecturalFraming.topology}} (microservices elevates traces from optional to recommended) and architecturalFraming.scaleTarget = {{architecturalFraming.scaleTarget}} (production-scale and above). OpenTelemetry (traces.otel) is the vendor-neutral instrumentation standard — instrument once, route to any compatible backend. OTel auto-instrumentation libraries (for Node.js, Python, Go, Java, .NET) provide zero-code trace context propagation for most frameworks. Trace backend choices: Jaeger / Zipkin (self-hosted, open-source), Honeycomb (SaaS, column-oriented, designed for high-cardinality traces), Datadog APM (SaaS, integrated with metrics and logs), AWS X-Ray (native for AWS-hosted services), Grafana Tempo (open-source, integrates with Prometheus + Loki stack). Cross-link: traces complement metrics (Section 3) and logs (Section 5) — the three together are the observability triad.
5. Observability — Logs
logs.format:      {{runtimeOperations.logs.format}}
logs.aggregator:  {{runtimeOperations.logs.aggregator}}
logs.retention:   {{runtimeOperations.logs.retention}}
Structured logging (logs.format = json) is the baseline for any application with an aggregator — plain text logs are unsearchable at scale. Structured logs emit key-value pairs that log aggregators can index and filter. Log aggregator choices: Datadog Logs, AWS CloudWatch Logs, GCP Cloud Logging, Elasticsearch + Kibana (ELK), Grafana Loki (pairs well with Prometheus + Grafana metrics stack). Log retention (logs.retention) must meet compliance requirements: SOC 2 typically requires 1 year; HIPAA requires 6 years; PCI DSS requires 1 year online. Cross-link to security.auditRetention.window = {{security.auditRetention.window}} — application logs and audit logs may share the same aggregator but must have separate retention policies if compliance requires longer audit log retention.
6. Alerting & On-Call
alerting.tool:               {{runtimeOperations.alerting.tool}}
alerting.thresholdStrategy:  {{runtimeOperations.alerting.thresholdStrategy}}

onCall.tool: {{runtimeOperations.onCall.tool}}
Cross-link: security.sensitivityTier = {{security.sensitivityTier}} — alerting.tool must not be none when sensitivity tier is high. High sensitivity projects require alert-driven incident response, not log polling. alerting.thresholdStrategy choices: static thresholds (simple, brittle to traffic spikes), dynamic/anomaly-based (ML-driven, reduces false positives at scale), SLO-based burn-rate alerts (fires when the error budget is depleting faster than allowed — the gold standard for SLO-aligned alerting). On-call tool (onCall.tool) choices: PagerDuty, OpsGenie, VictorOps, or platform-native (AWS SNS + Lambda). Cross-link to security.ir = {{security.ir.runbookStyle}} — the incident-response runbook defined in Security maps onto the on-call rotation and escalation policy defined here. These two fields must be consistent: a runbook without an on-call tool is unexecutable; an on-call tool without a runbook produces panic.
7. SLO & Feature Flags
slo.metrics:     {{runtimeOperations.slo.metrics | join: ", "}}
slo.errorBudget: {{runtimeOperations.slo.errorBudget}}

featureFlags.provider:  {{runtimeOperations.featureFlags.provider}}
featureFlags.lifecycle: {{runtimeOperations.featureFlags.lifecycle}}
SLO fields are rendered when architecturalFraming.scaleTarget = {{architecturalFraming.scaleTarget}} is production-scale or enterprise — SLOs are the contract between the engineering team and stakeholders about acceptable reliability. slo.metrics should enumerate the user-facing reliability signals (availability, p50/p95/p99 latency, error rate) for each critical user journey. slo.errorBudget defines the allowable failure rate (e.g., 99.9% availability = 43.8 minutes/month downtime budget). SLO-based burn-rate alerting (Section 6) consumes the error budget as an alert threshold. Feature flags (featureFlags.provider) decouple deployment from feature release — a deployed feature hidden behind a flag can be enabled incrementally (canary), rolled back instantly without a deployment, and targeted to specific users or segments. featureFlags.lifecycle defines the governance policy for flag cleanup: long-lived flags (kill switches, A/B tests) vs short-lived (feature rollouts that are cleaned up post-launch). Stale flags are a common source of technical debt and conditional complexity.

Maintenance mode mechanism: feature-flag-based (requires featureFlags.provider from Section 7 to be set), infrastructure-level (load balancer rule, DNS failover), or application-level (middleware check against a config value). maintenanceMode.userUX defines what the user sees during maintenance — a static page, a graceful degradation notice, or a retry-later HTTP 503 response. Health checks: liveness probes verify the process is alive (k8s restarts on failure); readiness probes verify the service can accept traffic (k8s removes from load balancer pool on failure); deep probes exercise downstream dependencies (database, cache, external APIs) to detect partial failures before they cascade. Health check paths are consumed by auth.serviceAuth = {{auth.serviceAuth}} configuration — decide whether health endpoints require authentication (external probes usually cannot authenticate; internal deep checks should). Cross-link: security.ir.runbookStyle = {{security.ir.runbookStyle}} defines the runbook format; runbooks.storagePath and runbooks.templateStyle here define where they live and the operational content template. incidentProcess.severityLevels should map to the escalation and notification SLA defined in security.ir.notificationSLA — the severity taxonomy must be consistent across security and operations. incidentProcess.postmortemTemplate closes the reliability loop: every production incident (Sev-1 at minimum) should produce a blameless post-mortem that feeds improvements back into runbooks and alerting thresholds.
