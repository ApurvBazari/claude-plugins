3. Vulnerability Scanning
scanning.deps:          {{security.scanning.deps}}
scanning.sast:          {{security.scanning.sast}}
scanning.dast:          {{security.scanning.dast}}
scanning.containerScan: {{security.scanning.containerScan}}
scanning.cadence:       {{security.scanning.cadence}}
Dependency scanning (deps) is table stakes for any project with package dependencies. SAST (static analysis: CodeQL, Semgrep, Snyk Code) is recommended when sensitivityTier is elevated or higher. DAST (dynamic analysis: OWASP ZAP, Burp Suite automated scan) is recommended when sensitivityTier = high — it catches runtime injection and authentication bypass issues that SAST misses. Container scanning is relevant when architecturalFraming.topology includes containerised workloads.
4. Threat Model
threatModel: {{security.threatModel}}
Should align with sensitivityTier: none is appropriate only for standard tier; stride-lite (abbreviated STRIDE worksheet) is appropriate for elevated; formal (full STRIDE or PASTA threat model document) is required for high. Cross-inputs: auth.strategy = {{auth.strategy}} (determines which auth threat vectors apply), apiIntegration.externalServices = {{apiIntegration.externalServices | join: ", "}} (each external integration is a threat boundary).

Per-column PII encryption (perColumnForPII) is recommended when privacy.piiCategories = {{privacy.piiCategories | join: ", "}} is non-empty — particularly for financial and health categories. mTLS for service-to-service communication is recommended when architecturalFraming.topology = {{architecturalFraming.topology}} is microservices — all inter-service calls traverse a network boundary and should authenticate both peers. Database-level encryption at rest (dbDefault) is the baseline and is provided by most managed databases; app-managed encryption (appManaged) provides an additional layer independent of the DB provider.
6. Application Security
headers.cors:              {{security.headers.cors}}
headers.csp:               {{security.headers.csp}}
headers.xFrameOptions:     {{security.headers.xFrameOptions}}

inputValidation.policy:    {{security.inputValidation.policy}}
inputValidation.library:   {{security.inputValidation.library}}
CORS policy should restrict origins to known domains — * is only acceptable for fully public, unauthenticated APIs. CSP (Content Security Policy) prevents XSS by declaring which sources may load scripts, styles, and media. X-Frame-Options: DENY prevents clickjacking. Input validation must cover every entry point: HTTP request bodies, query parameters, path parameters, headers, and WebSocket messages. Assumes apiIntegration.style = {{apiIntegration.style}}: GraphQL APIs require additional input depth-limit and complexity controls to prevent denial-of-service via deeply nested queries.
7. Audit & Incident Response
auditRetention.window:        {{security.auditRetention.window}}
auditRetention.tamperEvidence: {{security.auditRetention.tamperEvidence}}

ir.runbookStyle:   {{security.ir.runbookStyle}}
ir.notificationSLA: {{security.ir.notificationSLA}}
Cross-link to runtimeOperations.incidentProcess for full IR process detail — security defines the runbook style and notification SLA here; runtime operations defines the on-call rotation, escalation paths, and tooling. Tamper-evident audit logs (append-only tables, write-once object storage, external SIEM) are required when sensitivityTier is elevated or high — an attacker who can delete audit logs can erase their tracks. Audit retention window must meet the longest applicable compliance requirement: SOC 2 typically requires 1 year; HIPAA requires 6 years; PCI DSS requires 1 year online + 3 years archived.
