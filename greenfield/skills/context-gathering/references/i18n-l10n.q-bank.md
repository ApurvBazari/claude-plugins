> **See also:** `frontend-architecture.q-bank.md`

## Q-bank

### I.Q1 — Target locales
- **type:** multi-select free-text (locale codes)
- **showInLight:** true
- **isRiskCapture:** false
- **Prompt:** "Which locales must this app support at launch? Use BCP-47 codes (e.g., 'en-US', 'es-ES', 'ja-JP', 'ar-EG'). Always include the source locale. Pick the smallest set you can credibly ship — every locale you add multiplies translation cost + QA surface."
- **Stores to:** `phases.i18nL10n.targetLocales[]`
- **Default:** `["en-US"]` — source locale only. v1 cuts almost always ship monolingual.

### I.Q2 — Translation source
- **type:** single-select
- **options:** `["manual", "ai-assisted", "hybrid", "none"]`
- **showInLight:** true
- **isRiskCapture:** false
- **Prompt:** "How will copy be translated? (manual = professional translators / LSPs; ai-assisted = LLM-translated then human-reviewed; hybrid = AI first-pass + human polish for high-traffic strings; none = source-locale-only, no translations.)"
- **Stores to:** `phases.i18nL10n.translationSource`
- **Default:** `"none"` when `targetLocales.length === 1`; `"hybrid"` otherwise (best cost/quality trade-off for v1).

### I.Q3 — i18n library
- **type:** single-select
- **options:** `["next-intl", "react-i18next", "formatjs", "lingui", "native-intl", "none"]`
- **showInLight:** true
- **isRiskCapture:** false
- **Prompt:** "i18n library? (next-intl = Next.js App Router native, server-component-aware; react-i18next = mature React-first ecosystem; formatjs = ICU MessageFormat reference impl; lingui = type-safe macros + compile-time extraction; native-intl = browser `Intl.*` only; none = no library, source-locale text inlined.)"
- **Stores to:** `phases.i18nL10n.library`
- **Default:** `"next-intl"` when framework is Next.js App Router; `"react-i18next"` for other React stacks; `"none"` when `targetLocales.length === 1`.

### I.Q4 — Message file format
- **type:** single-select
- **options:** `["json", "po", "xliff", "yaml"]`
- **showInLight:** false
- **isRiskCapture:** false
- **Prompt:** "Translation file format? (json = simplest, library-native for next-intl/react-i18next; po = gettext, LSP-friendly + tooling-rich; xliff = industry-standard XML, best for enterprise TMS handoff; yaml = lingui/Ruby-style, comment-friendly.)"
- **Stores to:** `phases.i18nL10n.fileFormat`
- **Default:** `"json"` — matches the default library defaults and ships with no extra tooling.

### I.Q5 — RTL support
- **type:** yes/no
- **showInLight:** true
- **isRiskCapture:** false
- **Prompt:** "Support right-to-left scripts (Arabic, Hebrew, Persian, Urdu)? (yes = `dir=rtl` switching + logical CSS properties + bidi-aware components; no = LTR-only, easier but locks out ~500M users.)"
- **Stores to:** `phases.i18nL10n.rtlSupport`
- **Default:** `true` when `targetLocales` contains any of `["ar-*", "he-*", "fa-*", "ur-*"]`; `false` otherwise.

### I.Q6 — Date / number formatting
- **type:** single-select
- **options:** `["intl-api", "library-helper", "custom"]`
- **showInLight:** false
- **isRiskCapture:** false
- **Prompt:** "Date / number / currency formatting strategy? (intl-api = browser `Intl.DateTimeFormat` / `Intl.NumberFormat` directly; library-helper = `formatDate`/`formatNumber` from your i18n library; custom = wrap a third-party like date-fns or dayjs.)"
- **Stores to:** `phases.i18nL10n.dateNumberFormatting`
- **Default:** `"library-helper"` when `library !== "none"`; `"intl-api"` otherwise.

### I.Q7 — Plural rules
- **type:** yes/no
- **showInLight:** false
- **isRiskCapture:** false
- **Prompt:** "Handle CLDR plural rules properly (not just `count === 1`)? (yes = ICU MessageFormat with `{count, plural, one {...} few {...} many {...} other {...}}`; no = English-style binary `singular/plural` only, breaks in Slavic/Arabic locales.)"
- **Stores to:** `phases.i18nL10n.pluralRules`
- **Default:** `true` when `library !== "none"` and `targetLocales.length > 1`; `false` otherwise.

### I.Q8 — Content translation flow
- **type:** short-text
- **showInLight:** false
- **isRiskCapture:** false
- **Prompt:** "How will translations move between code and translators? (e.g., 'CSV export to Crowdin, weekly sync', 'Lokalise CLI + GitHub Action on PR merge', 'manual JSON edits — no TMS', 'Tolgee in-context editor'.)"
- **Stores to:** `phases.i18nL10n.contentTranslationFlow`
- **Default:** `""` when `translationSource === "none"`; `"manual JSON edits"` when `targetLocales.length <= 2`; suggest a TMS workflow otherwise.

### I.Q9 — Bundle delivery
- **type:** single-select
- **options:** `["bundled", "cdn", "lazy"]`
- **showInLight:** false
- **isRiskCapture:** false
- **Prompt:** "How are message bundles delivered to the client? (bundled = inlined in the JS bundle, simplest but bloats initial load per locale; cdn = static JSON files on CDN, cached + versioned; lazy = dynamic import per-locale, ships only the active locale.)"
- **Stores to:** `phases.i18nL10n.delivery`
- **Default:** `"lazy"` when `targetLocales.length > 3`; `"bundled"` otherwise (overhead trivial for ≤3 locales).

### I.Q10 — Static vs dynamic text
- **type:** single-select
- **options:** `["static", "dynamic", "hybrid"]`
- **showInLight:** false
- **isRiskCapture:** false
- **Prompt:** "Where does translated copy live? (static = all strings in message catalogs at build time; dynamic = user-generated content needs runtime translation/MT; hybrid = static UI copy + dynamic content fields like product descriptions.)"
- **Stores to:** `phases.i18nL10n.textType`
- **Default:** `"static"` for v1; flip to `"hybrid"` if domain model includes user-generated content surfaced cross-locale.

### I.Q11 — SEO hreflang
- **type:** yes/no
- **showInLight:** true
- **isRiskCapture:** false
- **Prompt:** "Emit `<link rel=alternate hreflang=...>` tags + locale-prefixed URLs (e.g., `/es/about`)? (yes = required for Google to serve the right locale per region; no = single canonical URL, fine for behind-login apps.)"
- **Stores to:** `phases.i18nL10n.seoHreflang`
- **Default:** `true` when `targetLocales.length > 1` and the app has any public-indexable pages; `false` for behind-login dashboards.

## Risk capture

- **type:** bulleted free-text
- **isRiskCapture:** true
- **showInLight:** true
- **tagSuggestions:** ["missing-translation", "rtl-layout-break", "plural-mismatch", "hreflang-gap", "encoding-mojibake", "locale-fallback-loop", "tms-sync-drift", "currency-rounding"]

- **Stores to:** `phases.i18nL10n.qRisks[]` + appends to top-level `risks[]` with `phase: "i18nL10n"`
